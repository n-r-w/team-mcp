# Disk storage layout

This document describes the on-disk files and directories generated by Team MCP under `TEAM_MCP_MESSAGE_DIR`.

## Scope

- This document covers the runtime storage layout for desks, topics, messages, lookups, and payloads.
- It describes generated runtime artifacts only. It does not describe source code layout or internal Go package structure.

## Summary

- Team MCP stores each desk in its own directory under `TEAM_MCP_MESSAGE_DIR`.
- The root directory also contains two global lookup directories: `_topics` and `_messages`.
- Each desk keeps a readable mirror of the latest state and a committed version snapshot for the same state.
- Message payloads are stored as separate Markdown files, not embedded into desk metadata.

## Root directory

```txt
<TEAM_MCP_MESSAGE_DIR>/
├── _messages/
├── _topics/
└── <desk_id>/
```

- `_topics/` stores one lookup file per topic. Each file maps `topic_id -> desk_id`.
- `_messages/` stores one lookup file per message. Each file maps `message_id -> desk_id + topic_id`.
- `<desk_id>/` stores all authoritative artifacts for one desk.

## Desk directory

```txt
<TEAM_MCP_MESSAGE_DIR>/<desk_id>/
├── .versions/
│   └── <desk_version>.json
├── desk.json
├── topics/
│   ├── <topic_id>.json
│   └── <topic_id>.versions/
│       └── <topic_version>.json
└── <message_id>.md
```

- `desk.json` is the latest readable mirror of desk state.
- `.versions/<desk_version>.json` is the committed desk snapshot. The implementation keeps only the newest snapshot.
- `topics/<topic_id>.json` is the latest readable mirror of one topic state.
- `topics/<topic_id>.versions/<topic_version>.json` is the committed topic snapshot. The implementation keeps only the newest snapshot.
- `<message_id>.md` stores the Markdown payload of one message.

## File purpose

### Global lookups

- `_topics/<topic_id>.json`
  - Purpose: resolve a topic directly to its owning desk.
  - Content: `{ "desk_id": "<desk_id>" }`
  - Why it exists: `topic_id` lookups do not need to scan all desk directories.

- `_messages/<message_id>.json`
  - Purpose: resolve a message directly to its owning desk and topic.
  - Content: `{ "desk_id": "<desk_id>", "topic_id": "<topic_id>" }`
  - Why it exists: `message_get` does not need to scan all desks and topics.

### Desk state

- `desk.json`
  - Purpose: expose the latest desk state in one readable file.
  - Key fields:
    - `version`: current desk version.
    - `created_at`: desk creation timestamp in Unix nanoseconds.
    - `topics`: ordered list of topic headers.
    - `topic_by_title`: exact-title deduplication index inside the desk.

- `.versions/<desk_version>.json`
  - Purpose: store the committed desk state for versioned writes.
  - Why it exists: the server can detect conflicts and recover from a stale or broken mirror file.

### Topic state

- `topics/<topic_id>.json`
  - Purpose: expose the latest topic state in one readable file.
  - Key fields:
    - `version`: current topic version.
    - `messages`: ordered list of message headers.
    - `message_by_normalized_title`: deduplication index for normalized titles inside the topic.

- `topics/<topic_id>.versions/<topic_version>.json`
  - Purpose: store the committed topic state for versioned writes.
  - Why it exists: the server can publish topic changes safely and prune stale snapshots later.

### Message payload

- `<message_id>.md`
  - Purpose: store message content as Markdown.
  - Why it exists: payload stays separate from metadata and lookup indexes.

## What each tool generates

### `desk_create`

```txt
<root>/
├── _messages/
├── _topics/
└── <desk_id>/
    ├── .versions/
    │   └── 00000000000000000001.json
    ├── desk.json
    └── topics/
```

- `desk_create` creates the root lookup directories if they do not exist.
- `desk_create` creates one desk directory.
- `desk_create` writes `desk.json` and the first desk snapshot.
- `desk_create` does not create topic files or message files.

### `topic_create`

```txt
<root>/
├── _topics/
│   └── <topic_id>.json
└── <desk_id>/
    ├── desk.json
    ├── .versions/
    │   └── <next_desk_version>.json
    └── topics/
        ├── <topic_id>.json
        └── <topic_id>.versions/
            └── 00000000000000000001.json
```

- `topic_create` writes a global topic lookup file in `_topics/`.
- `topic_create` writes a topic mirror file and the first topic snapshot.
- `topic_create` updates `desk.json` and the desk snapshot because the desk now contains one more topic.
- Repeating `topic_create` with the same exact title inside the same desk returns the existing topic instead of creating duplicates.

### `message_create`

```txt
<root>/
├── _messages/
│   └── <message_id>.json
└── <desk_id>/
    ├── <message_id>.md
    └── topics/
        ├── <topic_id>.json
        └── <topic_id>.versions/
            └── <next_topic_version>.json
```

- `message_create` writes a global message lookup file in `_messages/`.
- `message_create` writes the Markdown payload file in the desk directory.
- `message_create` updates the topic mirror file and the topic snapshot.
- `message_create` does not update desk metadata because the desk composition does not change.
- Repeating `message_create` with the same normalized title inside the same topic returns `duplicate_title` and the existing `message_id`.

## Key behavior

- Lookup directories exist to make `topic_id` and `message_id` resolution direct and cheap.
- Mirror files and committed snapshots exist side by side. The mirror is easy to read. The snapshot is the committed source used for safe versioned writes.
- Topic order belongs to desk state. Message order belongs to topic state.
- Topic title deduplication is scoped to one desk.
- Message title deduplication is scoped to one topic and uses normalized titles.
- Message payload is stored separately from topic metadata to keep indexes small and direct lookups cheap.
- The server prunes older snapshots and keeps only the newest committed version for each desk and topic.

## Operational note

- `TEAM_MCP_MESSAGE_DIR` may be missing, empty, or already contain Team MCP runtime state from a previous run.
- Use a dedicated directory for Team MCP data.
- The directory becomes non-empty during runtime as soon as desks, topics, or messages are created.
